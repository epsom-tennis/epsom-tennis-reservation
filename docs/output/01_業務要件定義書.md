## このシステムを作る最大の理由

イベント参加者への連絡が手作業に依存しており、取り違えリスクと工数が課題。  
予約受付・応募完了通知・当落通知をLINEで自動化し、スタッフの連絡ミスをゼロにするために作る。

#### 1-2. イベント運営の現状

テニスイベントを定期的に開催しており、参加者の応募・当落管理を運用中。  
現状は以下の課題がある：

- 応募受付がGoogleフォームで完結しているが、完了通知が手作業
- 当落連絡もスタッフが個別にLINEで送信しており、工数・ミスが発生しやすい
- 既存のLINE公式アカウントに約200名の友だちがいるが、個人宛への確実な送信手段がない
- キャンセル連絡が来た際に「誰に何を送ったか」の履歴が残らない

### 3. このシステムが存在する理由（WHY）

「参加者への連絡を確実に・個人宛に・履歴を残しながら自動化することで、スタッフの手作業を削減し、取り違えゼロの運営体制を作る」

現状、応募完了通知・当落連絡はスタッフが手動で行っており、取り違えリスクと工数が課題。このシステムはその問題を解消するために作る。

### 6. 現状の業務（As-Is）

#### 6-1. 現状の課題

| # | 課題 | 詳細 |
|---|---|---|
| 1 | 応募完了通知が手作業 | フォーム送信後、スタッフが個別にLINEで連絡しており工数がかかる |
| 2 | 当落連絡の取り違えリスク | 手動でLINEを送るため、送り間違いの可能性がある |
| 3 | 個人への確実な送信手段がない | LINE一斉送信はできるが、特定の個人に確実に届けるUser IDの管理ができていない |
| 4 | 送信履歴が残らない | 誰に何を送ったかの記録がなく、キャンセル対応時に確認できない |

#### 6-2. 現状の業務フロー（As-Is）

```
参加者がGoogleフォームに入力
→ スタッフがフォームの回答を確認
→ スタッフが手動でLINEに応募完了メッセージを送信
→ 当落確定後、スタッフが手動で個別にLINEで当落連絡
→ キャンセル連絡が来ても記録がなく対応が煩雑
```

### 7. 新しい業務（To-Be）

#### 7-1. 実現したいこと

| # | 要件 | 詳細 |
|---|---|---|
| 1 | 受付コードで個人を確実に特定 | 同名の人がいても受付コードで1対1紐付けができる |
| 2 | フォーム送信と同時にLINE通知 | スタッフ操作不要で自動的に応募完了通知が届く |
| 3 | 当落連絡をボタン1つで全員に送信 | スプレッドシートで当選・落選を入力 → ボタン1つで全員へLINE送信 |
| 4 | 送信履歴がスプレッドシートに残る | 誰にいつ何を送ったかが全てスプレッドシートで確認できる |
| 5 | スタッフグループにもログが流れる | 送信のたびLINEグループに自動投稿されるので、キャンセル対応時にすぐ確認できる |

#### 7-2. 新しい業務フロー（To-Be）

```
【初回セットアップ（1回のみ）】
既存200名全員に受付コードをpush送信

【以降のイベントごと】
① 参加者がLINE公式アカウントに「応募」と送信
   → GAS Webhookが受信 → ランダム受付コードを生成・返信
   → User IDと受付コードをスプレッドシートに保存

② 参加者がGoogleフォームに入力（受付コード欄に届いたコードを入力）
   → フォーム送信トリガー → 受付コードでUser IDを突合
   → 個人宛に「応募完了！」LINE通知
   → スタッフグループに「田中太郎さんの応募受付完了 9:10」を自動投稿

③ 当落確定後
   → スプレッドシートの「結果」列に当選/落選を入力
   → ボタン1つ押す → 全員に個人宛LINE送信
   → スタッフグループに送信ログを一括投稿
```

## B. システム共通部分（次案件でも使い回せる）

### 10. 開発方針

| 項目 | 方針 |
|---|---|
| 実装基盤 | Google Apps Script（GAS）+ Googleスプレッドシート |
| 認証情報管理 | GASのスクリプトプロパティに保存（コード直書き禁止） |
| 引き継ぎ性 | CLAUDE.md・本定義書で文脈を保存。GASはシンプルに保つ |
| 開発ツール | Claude Code（VS Code拡張） |

### 11. 制約条件

| 制約 | 内容 |
|---|---|
| LINEメッセージ上限 | ライトプラン5,000通/月（イベント1回あたり最大100通） |
| GAS実行時間 | 1回の実行は6分以内（50名一括送信でも問題なし） |
| LINE User IDの取得 | getFollowers APIで既存フォロワーのUser IDを一括取得可能 |
| 既存フォームの変更 | 受付コード欄を1つ追加するだけ。それ以外は変更不要 |

### 12. 用語定義

| 用語 | 定義 |
|---|---|
| 受付コード | 参加者を一意に識別するためのランダム英数字コード。「応募」送信時に自動発行される |
| User ID | LINEが各ユーザーに割り当てる内部ID。push送信に必要。表示名とは異なる |
| push送信 | LINE Messaging APIを使い、特定のUser IDに対して送信するメッセージ |
| Webhook | LINEからのメッセージをGASが受信するためのHTTPエンドポイント |
| フォームトリガー | Googleフォームに回答が送信されたときにGASが自動実行される仕組み |
| 送信ログ | 誰にいつ何を送ったかの記録。スプレッドシートとスタッフLINEグループの両方に残る |
